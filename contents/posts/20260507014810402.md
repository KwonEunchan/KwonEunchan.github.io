### GitHub Page를 선택한 이유

사실 운영 및 관리 측면에서 GitHub Page는 그다지 선호되지 않는 블로그 플랫폼입니다.\
그저 정적 호스팅 하나만을 지원할 뿐, A부터 Z까지 스스로 모든 기능을 구현해야만 비로소 나만의 블로그를 오픈할 수 있으니까요.\
그치만 제게는 오히려 **'내가 원하는 기능을 내 손으로 구현할 수 있다'**&#xB294; 그 점이 매력적으로 다가왔습니다.

### GitHub Page를 선택했기에 고민해야만 했던 내용들

1. **개발 언어**

일반적으로는 Ruby 언어를 많이 사용한다고 합니다만, 저는 프론트엔드 개발을 취미 삼아 종종 해왔던지라 익숙한 React 프레임워크를 채용했습니다.\
최종 결과물에는 큰 영향을 미치지는 않았겠지만 최신 기술도 한 번 경험해보고 싶어 최종적으로는 Next.js를 이용하여 구현하게 되었습니다.

2. **게시물 관리**

사실 가장 크게 고민했던 부분입니다.\
블로그를 운영하자는 생각은 학부생 때부터 줄곧 해왔었고 종종 시도하기도 하였으나 게시물 관리의 어려움 탓에 Velog나 Tstory 등의 플랫폼을 고려하기도 했으니까요.\
DB 연동을 위해 서버를 하나 두자니 비용이 발생하고, 매번 마크다운 에디터로 작성한 파일을 푸시하기엔 너무 번거로울 것 같았습니다.

그래서 제가 처음에 고안했던 것은 Notion과 연동하여 해당 워크스페이스의 게시물을 내 블로그에서 보여주는 방식이었습니다.\
그렇게 되면 서버를 따로 둘 필요도 없으니 비용도 아끼고, 강력한 마크다운 에디터를 별도로 구현할 필요 없이 사용할 수 있었으니까요.\
처음엔 그렇게 블로그를 오픈했었지만.. Notion API는 100건 이상의 데이터는 한 번의 요청에서 전송해주지 않는다는 것을 뒤늦게 알게 되어 곤란했습니다.

이에 새롭게 고안한 것이 블로그 페이지에 마크다운 에디터를 심어놓고, **작성한 파일을 즉석에서 푸시하는 방식**입니다.\
다행히 Notion 못지 않게 강력한 에디터인 **'BlockNote'**&#xB97C; 알게 되어 비교적 쉽게 에디터 구현은 완료가 되었습니다.\
문제는 파일을 푸시하기 위해서는 GitHub Token이 필요한데.. 오픈된 저장소에 토큰을 하드 코딩하여 게시하기는 조금 리스크가 있어 어려움이 있었습니다.\
그러나 **게시물을 작성할 때&#x20;****브라우저에서 토큰 값을 입력하도록 구현하면 되겠다****는 아이디어가 떠올라 블로그 화면에서 작성한 마크다운 파일을 바로 푸시할 수 있게 되었습니다.**

3. **빌드 자동화**

게시물을 새롭게 작성할 때마다 빌드가 되어야 블로그 화면에서 신규 게시물을 확인할 수 있습니다.\
이에 GitHub Action을 알게 되어 변화가 감지되면 빌드를 수행하도록 처리해두었습니다.

그러나 한 게시물에 다수의 이미지가 포함되는 경우가 문제가 되었습니다.\
개별 이미지가 각각 푸시되는 탓에 빌드가 여러 번 수행되었고, 중복이 발생하여 실패하는 케이스가 종종 발생했기 때문입니다.\
\
이에 **커밋은 누적된다는 점에서 아이디어를 착안하여, 새롭게 빌드가 발생하면 이전 빌드 동작을 모두 취소하도록 처리**하였습니다.\
어차피 마지막 빌드에서 이전에 수행된 내용을 모두 포함하고 있기에 최종적인 빌드 내용에는 문제가 전혀 없었습니다.
